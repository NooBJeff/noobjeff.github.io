Linux: 我TM错爆

昨天和今天都花在了配置服务器环境上。为了装个nginx+php+mysql我特么竟然卡了2天（忘了用lnmp全家桶）。

服务器运行的是CentOS7，首先为yum添加remi repo，下载php，nginx等软件。这一步花的时间不多。

软件都下好了便开始痛苦的配置了。。。

nginx

以前用的都是Apache，因此这次是我第一次配置nginx，很痛苦。
首先创建专门的用户及用户组www www，设置好 Document Root 的所属用户及权限，设置好 nginx 进程运行的用户。设置完这些应该就差不多了。我便打开浏览器一看：

403 Forbidden

百度谷歌了一下午加一晚上。最后终于发现，是一个叫什么 SELinux 原因。最最讽刺的是，这个解决方法来自于 StackOverflow 上搜索nginx 403第一个提问的第二个回答。所以这本该是一次快速地查找答案并修复的过程，最后花了我两天。

下次有问题一定先查 StackOverflow

php-fpm

blank page

解决完nginx 403 的问题，我心中一阵舒爽，以为前方的道路已经通畅，胜利就在不远方。然而，本该配置好的php又出了问题

似乎现在php运行的工作都用一个叫 php-fpm 的东西来进行类似进程管理的调度。于是我也用了这么个东西，但是运行的时候，在浏览器访问php页面，得到的结果永远是一个空白页面，什么都没有，status code却是正常的200。这就很奇怪。于是又百度谷歌爆栈搜索半天，发现是nginx的配置文件没有include fastcgi_params，遂加上。果然，不再是空白页面了–是更恼人的

php Permission Denied

既然是权限问题，那我就去查看各个文件的用户及权限是否正确。我发现问题出在php-fpm 进程的运行用户上，似乎php-fpm 的默认运行用户是Apache，于是我打开php-fpm.conf，像nginx.conf里一样加上 user=www。保存退出，重新运行php-fpm，居然 TMD 报错了，不认 user 这个配置选项。那我要 TM 怎么改你的运行用户？

于是又百度谷歌爆栈搜索了半天，终于发现配置用户需要到另一个文件 www.conf里去设置，我的天啊，连 php 的官方 manual 上都写的是往 php-fpm.conf 里加 user（点此有真相 ）。这解决方法谁能料得到，坑，太坑了。

于是又双叒叕配置了 php-fpm 的运行用户为www，重启，进网页，ok，终于能看到内容了：

Could not found driver

这个问题是在php代码中使用PDO的时候出现的，问题比较简单：没装PDO，装上php的mysql插件就好了。

好嘛，都解决到最后一环：数据库了。只要数据库调通，这整个网站也就基本能用了。但是。。。。。

Access denied for user ‘xxx’@’localhost’ (using password: YES)

去你妈的权限问题

权限权限权限，我感觉自己遇到的问题中70%都和Linux的权限设置有关。

哦等一下，黑错了，这里是MySQL的用户权限问题，这锅Linux不背。。。

还是从头讲一下我认为的MySQL用户权限方面的通常解决方法吧。

select * from mysql.user;

可查看用户信息及权限

首先删掉 mysql.user 表中的无名用户''，因为在某些情况下这个无名用户可能造成困扰，而且他似乎也没什么用，先删了省得后面出问题。

接下来创建用户，并用 GRANT 语句赋予该用户在某些数据库上的某些权限，GRANT 语句中的 TO 的对象可以是’username’@’localhost’或者@’%’，localhost限定用户只能本地访问数据库，% 可允许外网访问。

但是我这边用 GRANT 加 FLUSH PRIVILEGES 并不管用，用户的权限并没有更新。所以我选择手动更新 mysql.user 表中的权限，修改完后发现终于一切都正常了（至少表面上是这样）。

至此nginx + php + mysql 终于都配置好了，聊天室应用也得以正常的运行。于是我打开浏览器开始实际测试应用的各个功能是否能够正常的运行。

万万没想到，还是有坑。

Permission Denied

TMD 怎么又是权限问题？

这个问题出在图片上传这个功能上。

当我网页上用 uploadify 进行图片上传的时候，服务器一直响应 500 内部错误。但是这个功能代码上肯定是没问题的，因为在测试服务器上运行的时候一切正常。

遂查看 nginx 及 php 的 error log，具体忘了是哪个log了，总之log里写的是“处理post（上传图片）请求时，往xxxx文件夹写入失败”。查了一下，似乎HTTP POST 请求内容较大的时候，服务器会创建缓存保存POST相关内容，但是如果没有创建缓存的文件夹的相应权限，那自然就失败报错 500 了。于是我又 chown www:www /path/to/cache，再次测试，终于，终于好了。
